## 2. 微服务详细划分

### 2.1 Auth Service (认证授权服务)

**职责边界**:
- 用户注册/登录（手机号/邮箱）
- Web3 钱包地址绑定与验证
- JWT Token 签发与刷新
- SIWE (Sign-In with Ethereum) 签名验证
- 可选 KYC 流程管理

**核心功能**:
```go
- POST   /api/v1/auth/register          // 用户注册
- POST   /api/v1/auth/login             // 账号密码登录
- POST   /api/v1/auth/wallet/bind       // 绑定钱包地址
- POST   /api/v1/auth/wallet/login      // 钱包签名登录
- POST   /api/v1/auth/token/refresh     // 刷新 Token
- POST   /api/v1/auth/kyc/submit        // 提交 KYC 资料
- GET    /api/v1/auth/kyc/status        // 查询 KYC 状态
```

**技术实现**:
- 语言：Go 1.22+
- 框架：Gin
- 数据库：PostgreSQL (users, wallets, kyc_records 表)
- 缓存：Redis (session, blacklist)
- 依赖：golang-jwt/jwt, ethereum/go-ethereum (SIWE 验证)

**事件发布**:
- `user.created` - 用户创建成功
- `wallet.bound` - 钱包绑定成功
- `kyc.approved` - KYC 审核通过

#### 2.9.5 技术实现细节

**Python FastAPI 示例**:
```python
from fastapi import FastAPI, BackgroundTasks
from langchain.chat_models import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

app = FastAPI()

llm = ChatOpenAI(model="gpt-4", temperature=0.7)

@app.post("/api/v1/ai/memory/summary")
async def generate_memory_summary(request: MemoryRequest):
    prompt = ChatPromptTemplate.from_template("""
    你是一位精通五行文化的 AI 助理。
    请为以下记忆生成 50 字以内的诗意描述：
    
    内容：{content}
    五行属性：{element}
    时间：{time}
    地点：{location}
    """)
    
    chain = prompt | llm
    response = await chain.ainvoke({
        "content": request.content,
        "element": request.element,
        "time": request.time,
        "location": request.location
    })
    
    return {"summary": response.content}

@app.post("/api/v1/ai/chat")
async def chat_stream(request: ChatRequest):
    # SSE 流式响应
    async def generate():
        async for chunk in llm.astream(request.message):
            yield f"data: {chunk.content}\n\n"
    
    return StreamingResponse(generate(), media_type="text/event-stream")
```

## 3. 服务间通信机制

### 3.1 同步通信 (REST/gRPC)

| 调用场景 | 协议 | 说明 |
|----------|------|------|
| 前端→后端 API | REST/HTTP | 标准 RESTful 接口 |
| AI 服务→其他服务 | gRPC | 高性能内部调用 |
| 服务→服务实时查询 | REST/HTTP | 简单场景使用 |

### 3.2 异步通信 (NATS 消息队列)

**事件总线设计**:
```
发布者 (Publisher) → NATS Server → 订阅者 (Subscriber)

主题命名规范:
- entity.action (例：user.created, memory.deleted)
- 服务名.事件名 (例：ai.summary.generated)
```

**核心事件流**:
```
1. 用户注册流程:
   Auth Service → user.created → [User Service, AI Service]

2. NFC 绑定流程:
   Device Service → device.bound → [NFT Service]
   NFT Service → nft.minted → [Device Service, User Service]

3. 记忆 AI 生成流程:
   Memory Service → memory.created → [AI Service]
   AI Service → ai.summary.generated → [Memory Service]

4. 购买流程:
   Market Service → order.created → [Payment Service]
   Payment Service → payment.completed → [Market Service, NFT Service]
```

### 3.3 WebSocket 实时推送

**使用场景**:
- AI 对话流式输出

- 交易状态变更通知

## 4. 数据一致性保障

### 4.1 分布式事务策略

**最终一致性模型**:
- 不采用强一致性分布式事务（如 2PC）
- 使用 Saga 模式或事件溯源保证最终一致性
- 关键业务（支付、NFT 转移）需要补偿机制

**Saga 模式示例（购买流程）**:
```
1. Market Service: 创建订单 (pending)
2. Payment Service: 扣款
   - 成功 → 发布 payment.completed
   - 失败 → 发布 payment.failed → Market Service 取消订单
3. Market Service: 订单完成 → 发布 order.completed
4. NFT Service: 转移 NFT 所有权
```

### 4.2 幂等性保证

**实现方式**:
- 所有写操作接口要求携带 `idempotency_key`
- Redis 存储 key-value，TTL 24 小时
- 重复请求直接返回首次执行结果

## 6. 监控与可观测性

### 6.1 指标收集 (Metrics)

**每个服务必须暴露的指标**:
- HTTP 请求延迟 (p50, p95, p99)
- 错误率
- 数据库连接池状态
- 缓存命中率
- 消息队列积压量

**技术栈**: Prometheus + Grafana

### 6.2 分布式追踪 (Tracing)

**实现方案**: Jaeger / Zipkin
- 每个请求携带 `trace_id`
- 跨服务调用自动传播 trace context
- 关键业务链路可视化

### 6.3 日志聚合 (Logging)

**技术栈**: ELK Stack (Elasticsearch, Logstash, Kibana)
- 结构化日志 (JSON 格式)
- 统一日志字段：timestamp, level, service, trace_id, user_id
- 敏感信息脱敏
